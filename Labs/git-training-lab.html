🌱 Getting Started

Exercise I — Adding Files

Set up your identity, create a bare remote repo, clone it, and make your first commit.

Configure your Git identity. This will appear on every commit.
Terminal
$ git config --global user.name "YourUSERNAME" $ git config --global user.email "YourEMAIL" $ git config --global init.defaultBranch main # Verify your settings: $ git config --global -l
Create a bare shared repository (this acts as your "remote origin").
Terminal
$ git init --bare --shared ~/git_remote_repo
💡 A bare repo has no working directory — it's only the .git internals. Perfect for a shared central server.
Clone it into Harry's workspace. Then explore the .git folder.
Terminal
$ git clone ~/git_remote_repo ~/harry $ cd ~/harry
Create two files with the content shown below.
📄 main_e.txt
1include library.txt 2Hello World
📄 library.txt
1library line 1 2library line 2 3library line 3 4library line 4 5library line 5
Stage, review, and commit your first changes.
~/harry
~/harry]$ git add * ~/harry]$ git status ~/harry]$ git commit -m "Initial project"
📁 Getting Started

Exercise II — Restructuring Files

Rename and reorganize files using git mv, then push to the remote.

Rename main_e.txt → main.txt, create a libs/ folder, and move library.txt into it.
~/harry
~/harry]$ git mv main_e.txt main.txt ~/harry]$ mkdir libs ~/harry]$ git mv library.txt libs/
Update main.txt to reference the new path, then stage the change.
📄 main.txt MODIFIED
1include libs/library.txt 2Hello World
~/harry
~/harry]$ git add -u
Review, commit, and push the restructuring.
~/harry
~/harry]$ git status ~/harry]$ git diff --staged main.txt ~/harry]$ git commit -m "Restructuring project" ~/harry]$ git push
✅ Always use git diff --staged before committing — it shows exactly what's about to be committed.
👥 Collaboration

Exercise III — Teamwork

Simulate two developers (Harry & Sally) working on the same repo. Learn push/pull with rebase.

In Harry's repo, change line 3 of libs/library.txt, then commit.
📄 libs/library.txt — Harry's edit
1library line 1 2library line 2 3Harrys change *here* 4library line 4 5library line 5
~/harry
~/harry]$ git commit -a -m "Harrys 1st changes in our library"
Clone the remote to create Sally's workspace before pushing Harry's changes.
~/harry
~/harry]$ git clone ~/git_remote_repo ~/sally ~/harry]$ git push
💡 Sally cloned before Harry pushed — she doesn't have Harry's latest changes yet.
In Sally's repo, change line 1, commit, and try to push.
📄 libs/library.txt — Sally's edit
1Sallys 1st change 2library line 2 3library line 3 4library line 4 5library line 5
~/sally
$ cd ~/sally ~/sally]$ git commit -a -m "Sallys 1st changes in our library" ~/sally]$ git push ! [rejected] main -> main (non-fast-forward)
⚠️ Push fails! The remote has Harry's new commits that Sally doesn't have. She needs to pull first.
Pull with rebase, verify, then push.
~/sally
~/sally]$ git pull --rebase ~/sally]$ git log ~/sally]$ git diff HEAD^ HEAD # see both changes ~/sally]$ git push
Sync Harry's repo with Sally's latest changes.
~/harry
$ cd ~/harry ~/harry]$ git pull --rebase
⚡ Collaboration

Exercise IV — Conflicts (with Rebase)

Both developers edit the same line. Learn to resolve merge conflicts during a rebase.

Harry changes the last line (line 5) of libs/library.txt, commits, and pushes.
~/harry
# Change line 5 to: "another of Harrys changes" ~/harry]$ git commit -a -m "Harry's 2nd changes in our library" ~/harry]$ git push
Sally changes the same last line, commits, and tries to push.
~/sally
$ cd ~/sally # Change line 5 to: "Sallys 2nd change.." ~/sally]$ git commit -a -m "Sally 2nd change in library" ~/sally]$ git push ! [rejected] — remote has changes you don't have
Pull with rebase — this time you'll hit a conflict.
~/sally
~/sally]$ git pull --rebase CONFLICT (content): Merge conflict in libs/library.txt
🔥 Both developers changed the same line — Git can't auto-merge. You must resolve it manually.
Open the file and resolve the conflict markers.
Sally's first change
Library line 2
Harry's change here
Library line 4
<<<<<<< HEAD
Another of Harrys change =======
Sally 2nd change >>>>>>> ad9e470 (+Sally 2nd change)
✏️ Delete the conflict markers and keep the content you want — one line, both lines, or a rewrite.
Mark the conflict resolved and continue the rebase.
~/sally
~/sally]$ git add libs/library.txt ~/sally]$ git status rebase in progress; all conflicts fixed: run "git rebase --continue" ~/sally]$ git rebase --continue ~/sally]$ git log -p -2 ~/sally]$ git push
🏷️ Intermediate

Exercise V — Tagging

Create annotated tags to mark releases and sync them across repositories.

Create and push a release tag from Sally's repo.
~/sally
~/sally]$ git tag Release_01 -m "Official Release_01 GA" ~/sally]$ git log --graph --oneline --all ~/sally]$ git push --tags
Fetch the tag into Harry's repo.
~/harry
$ cd ~/harry ~/harry]$ git log --graph --oneline --all # no tag yet! ~/harry]$ git fetch --tags ~/harry]$ git log --graph --oneline --all # now you see it ~/harry]$ git rebase
* 9ec4c94 (HEAD → main, tag: Release_01, origin/main) + Sally 2nd change * 9c8fe9e +Another of Harrys change * 39da746 +Sally's first change * 0a09798 +Harry's change here
🔀 Intermediate

Exercise VI — Branching

Create a feature branch, push it with upstream tracking, and continue work on main.

Create and check out a bugfix branch.
~/harry
~/harry]$ git branch bugfix/release_01_fix1 ~/harry]$ git checkout bugfix/release_01_fix1
Make changes to both files, commit, and push with upstream tracking.
📄 main.txt
1include libs/library.txt 2Hello World *bugfixed*
📄 libs/library.txt
…... 6*bugfixing in lib*
~/harry (bugfix/release_01_fix1)
~/harry]$ git commit -a -m "bugfixing" ~/harry]$ git push --set-upstream origin bugfix/release_01_fix1 ~/harry]$ git branch -vv
Switch back to main and continue development there too.
~/harry
~/harry]$ git checkout main # Add a line to libs/library.txt: "Further development in master" ~/harry]$ git commit -a -m "making progress in the main" ~/harry]$ git log --graph --oneline --branches ~/harry]$ git push
🔗 Intermediate

Exercise VII — Merging

Merge the bugfix branch back into main and inspect the results with blame.

Merge the bugfix branch into main.
~/harry (main)
~/harry]$ git status # On branch main — working tree clean ~/harry]$ git merge bugfix/release_01_fix1 # Save the merge commit message
Inspect the merge and push.
~/harry
~/harry]$ git log --graph --oneline --all # see the merge ~/harry]$ git push ~/harry]$ git blame ./libs/library.txt
🔍 git blame shows who changed each line and when — invaluable for debugging.
🍒 Advanced

Exercise VIII — Checkout & Cherry-pick

Check out an old commit by tag, create a new branch from it, and cherry-pick a specific commit.

Check out the Release_01 tag (enters detached HEAD).
~/harry
~/harry]$ git checkout main ~/harry]$ git log --decorate ~/harry]$ git checkout Release_01 Note: checking out 'Release_01'. You are in 'detached HEAD' state.
⚠️ Detached HEAD means you're not on any branch. Create a branch if you want to keep new commits!
Create a new branch from this tag and cherry-pick a commit from main.
~/harry
~/harry]$ git checkout -b bugfix/release_01_fix2 ~/harry]$ git log main --decorate --name-only # Copy the SHA1 of "making progress in the main" commit ~/harry]$ git cherry-pick <SHA1> ~/harry]$ git diff HEAD^ HEAD ~/harry]$ git log --graph --oneline --all ~/harry]$ git push --set-upstream origin
⏪ Advanced

Exercise IX — Git Reverts of All Kinds

The most important safety net in Git. Learn to undo changes at every level — working dir, staged, committed (local), and pushed (remote).

Undo working directory changes — discard uncommitted edits.
~/harry (main)
~/harry]$ git checkout main # Edit main.txt — add "# Add temp line" ~/harry]$ git diff # ALWAYS review before discarding! ~/harry]$ git restore main.txt
🚨 git restore is irreversible — the changes are gone forever. Always git diff first.
Undo staged changes — unstage, then discard.
~/harry
# Edit main.txt again, then stage it: ~/harry]$ git add main.txt ~/harry]$ git diff --staged # review what's staged ~/harry]$ git restore --staged main.txt # unstage ~/harry]$ git restore main.txt # discard
Undo a local commit — reset to the previous commit.
~/harry
# Edit, commit: ~/harry]$ git commit -a -m "Adding temp line number 3 commit" ~/harry]$ git log --name-status ~/harry]$ git reset --hard HEAD^ ~/harry]$ git reflog # the commit still exists in reflog!
⚠️ --hard resets branch, index, and working tree. The commit is unreferenced but not deleted — git reflog can find it.
Revert a pushed commit — safely undo a commit that's already shared.
~/harry
# Reset back to the temp commit & push it: ~/harry]$ git reset --hard <SHA1> ~/harry]$ git push # Now revert it (creates a NEW commit that undoes it): ~/harry]$ git revert HEAD ~/harry]$ git push
💡 git revert is the safe way to undo pushed commits — it creates a new commit, preserving history.
Force-replace a remote branch — reset a branch and force push.
~/harry
~/harry]$ git checkout bugfix/release_01_fix1 ~/harry]$ git reset --hard HEAD~1 # Allow non-fast-forward on the bare repo: ~/harry]$ git -C ~/git_remote_repo config receive.denyNonFastForwards false ~/harry]$ git push --force-with-lease
🚨 --force-with-lease is safer than --force — it checks that no one else pushed in the meantime.
📦 Advanced

Exercise X — Git Stash

Stash work-in-progress, fix an urgent bug, then pop your stash back.

Start a regular task, then stash it when an emergency hits.
~/harry (main)
~/harry]$ git checkout main # Edit main.txt — add "Add regular task" ~/harry]$ git stash Saved working directory and index state WIP on main: ...
Do the emergency fix and commit it.
~/harry
# Edit main.txt — add "Add quick fix immediately" as first line ~/harry]$ git commit -a -m "Fix a bug in a quick"
Pop your stash and continue where you left off.
~/harry
~/harry]$ git stash pop Auto-merging main.txt # Resolve conflicts if any, then continue: ~/harry]$ git diff ~/harry]$ git commit -a -m "Regular task" ~/harry]$ git push
🩹 Advanced

Exercise XI — Format & Apply Patch

Create a patch file from a commit and apply it in another repository — useful for sharing changes without push access.

Add a line to Harry's library, commit, and format a patch.
~/harry/libs
$ cd ~/harry/libs # Add "patch line" to the end of library.txt ~/harry]$ git commit -am "harrys commit for patch" ~/harry/libs]$ git format-patch -1 main 0001-harry-s-commit-for-patch.patch
In Sally's repo, check and apply the patch.
~/sally/libs
$ cd ~/sally/libs ~/sally/libs]$ git pull ~/sally/libs]$ git apply --stat ~/harry/libs/0001-*.patch ~/sally/libs]$ git apply --check ~/harry/libs/0001-*.patch ~/sally/libs]$ git am --signoff < ~/harry/libs/0001-*.patch Applying: harry's commit for patch
✅ --signoff adds a "Signed-off-by" line — useful for tracking who approved/applied the patch.
🔬 Advanced

Exercise XII — Squash Commits

Use interactive rebase to squash multiple commits into one clean commit.

In Sally's repo, make 2 small commits that you want to combine.
~/sally
# Make 2 changes with 2 separate commits ~/sally]$ git log --pretty=oneline -3 9fb9d06 Add 2nd commit for squash 5822495 Add 1st commit line d3eb9fd harry's commit for patch
Run interactive rebase and squash the second commit into the first.
~/sally
~/sally]$ git rebase -i HEAD^^
📝 Interactive rebase editor
pick 5822495 Add 1st commit line s 9fb9d06 Add 2nd commit for squash # change "pick" → "s" (squash)
Save the combined commit message, verify, and push.
~/sally
Successfully rebased and updated refs/heads/main. ~/sally]$ git log -p -2 # now 1 commit instead of 2 ~/sally]$ git reflog # old commits are still in reflog ~/sally]$ git push
🌐 Advanced

Exercise XIII — Fetch from Another Repo

Fetch a branch directly from another developer's local repo — no origin needed.

From Sally's repo, fetch Harry's bugfix branch directly.
~/sally
~/sally]$ git branch * main ~/sally]$ git branch -r ~/sally]$ git fetch ~/harry bugfix/release_01_fix1:bugfix/release_01_fix1
Verify the fetched branch matches.
~/sally
~/sally]$ git log -1 --oneline bugfix/release_01_fix1 ~/sally]$ git -C ~/harry log -1 --oneline bugfix/release_01_fix1 # Both should show the same commit hash ~/sally]$ git checkout bugfix/release_01_fix1 ~/sally]$ git branch
